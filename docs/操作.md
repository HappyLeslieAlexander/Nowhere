# 运维指南

本指南涵盖了 Portal 启动后的关键运行行为。

## 启动

Portal 在接收流量之前会先验证配置。在 `net=mix` 模式下，TCP 和 UDP 监听器必须均成功绑定。进程会记录应用默认值后的实际启动 URL：

```text
portal::run: starting: portal://:<port>?tls=1&spec=auto&alpn=now/1&net=mix&dial=auto&rate=0&etar=0
```

在对比预期配置与运行时行为时，请参考此行输出。它显示的是实际生效的参数值，而不一定是原始命令行中输入的 URL 字符串。

## 日志记录

日志输出至标准输出（stdout），包含本地时间戳和严重级别。启动错误则输出至标准错误（stderr）。

可用的日志级别（通过 URL 参数设置）：

| 级别    | 行为                                     |
| ------- | ---------------------------------------- |
| `none`  | 禁用输出。                               |
| `debug` | 输出调试、信息、警告、错误及事件记录。   |
| `info`  | 输出常规运行日志、警告、错误及事件记录。 |
| `warn`  | 输出警告、错误及事件记录。               |
| `error` | 输出错误及事件记录。                     |
| `event` | 仅输出事件记录。                         |

若 `log` 参数值无法识别，则默认使用 `info` 级别。

## 事件记录

在启动时以及随后的每个 `NOW_REPORT_INTERVAL` 周期，若当前日志级别包含事件记录，Portal 就会输出一条检查点（checkpoint）记录：

```text
CHECK_POINT|MODE=0|PING=0ms|POOL=<n>|TCPS=<n>|UDPS=<n>|TCPRX=<bytes>|TCPTX=<bytes>|UDPRX=<bytes>|UDPTX=<bytes>
```

| 字段    | 含义                                        |
| ------- | ------------------------------------------- |
| `POOL`  | 等待首个请求帧的已认证 TLS/TCP 连接数。     |
| `TCPS`  | 活动 TCP 转发流数量。                       |
| `UDPS`  | 跨 QUIC DATAGRAM 和 UoT 的活动 UDP 流数量。 |
| `TCPRX` | 客户端至目标端的 TCP 字节数。               |
| `TCPTX` | 目标端至客户端的 TCP 字节数。               |
| `UDPRX` | 客户端至目标端的 UDP 载荷字节数。           |
| `UDPTX` | 目标端至客户端的 UDP 载荷字节数。           |

`MODE` 和 `PING` 在 v1 版本中保持固定。`NOW_REPORT_INTERVAL` 仅控制本地
记录的输出；它不会发送传输层保活数据包。

## 速率限制

`rate` 限制客户端至目标端的流量。`etar` 限制目标端至客户端的流量。
两者均为进程级限制，由活跃的 TLS/TCP 和 QUIC 会话共享。
对于 UoT，仅计算 UDP 载荷字节数；两字节的数据包长度和
建立连接时的帧开销不计入其中。

```text
portal://secret@:2077?rate=100&etar=200
```

换算关系如下：

```text
bytes_per_second = mbps * 125000
```

限制在传输层之上实施。它们不会选择或调整 QUIC
拥塞控制机制。

## QUIC 运行时行为

Portal（入口节点）：

- 启用 QUIC DATAGRAM；
- 在接受入站连接前要求进行 QUIC 重试（Retry）；
- 启动预认证连接时，包含一个双向流及 64 KiB 的连接级接收信用额度；
- 认证后将双向流数量限制提升至 `NOW_QUIC_MAX_STREMS`；
- 认证后将连接级接收信用额度提升至 32 MiB；
- 使用 16 MiB 的单流接收信用额度；
- 允许每个连接最多 32 MiB 未确认的流数据；
- 禁用单向流；
- 使用 BBR 拥塞控制算法；
- 使用 `NOW_UDP_IDLE_TIMEOUT` 作为 QUIC 空闲超时时间；以及
- 不发送 QUIC 传输层保活（keepalive）数据包。

QUIC DATAGRAM 的发送和接收缓冲区配置为 4 MiB。UDP 套接字请求 4 MiB 的发送和接收缓冲区（操作系统可能会限制或调整这些请求值）。

## UoT 运行时行为

只要启用了 TLS/TCP 监听器，UoT 即可使用；它没有独立的配置开关。在完成正常的传输层认证后，客户端会发送针对保留目标地址 `uot.nowhere.invalid:0` 的 TCP 请求、一个 UDP 目标设置帧，随后发送一系列带有双字节长度前缀的数据包。

每个 UoT 连接代表指向特定目标的一个逻辑 UDP 数据流。数据包边界得以保留，任一方向的流量都会刷新 `NOW_UDP_IDLE_TIMEOUT` 计时器，关闭 TLS/TCP 连接即会关闭该数据流。UoT 数据流会增加 `UDPS`、`UDPRX` 和 `UDPTX` 计数器，而不会增加 TCP 转发相关的计数器。

当 QUIC/UDP 无法穿透网络，但 TCP 队头阻塞（head-of-line blocking）问题又不可避免时，UoT 非常有用。若原生 UDP 传输可用，且需确保丢包不会导致无关数据包停滞，则应优先选择 QUIC DATAGRAM。 ## 认证截止时间

TLS 或 QUIC 握手完成后，认证过程使用一个基于以下公式计算得出的绝对截止时间：

```text
NOW_HANDSHAKE_TIMEOUT * [0.8, 1.2]
```

认证成功后立即继续后续流程。认证失败则会等待至上述截止时间。随后，TCP 连接在没有应用层响应的情况下关闭；QUIC 连接则以应用层代码 `0x01` 和原因 `access denied`（拒绝访问）关闭。

这种计时策略既限制了资源使用，又使得常见的认证失败场景难以被区分。

## 准入限制

在认证之前，Portal 实施一项由 TCP 和 QUIC 共享的进程级准入限制：

| 限制项                               | 数值  |
| ------------------------------------ | ----- |
| 认证前连接总数                       | `256` |
| 每个 IPv4 `/32` 或 IPv6 `/64` 的限制 | `32`  |

若 QUIC 连接尝试超过上述任一限制，该尝试将被忽略。若 TCP 连接超过上述任一限制，连接会被接受但随即关闭。一旦认证成功或失败，相应的连接槽位（slot）即被释放。

## 运行时控制参数

| 变量                    | 默认值  | 用途                                                 |
| ----------------------- | ------- | ---------------------------------------------------- |
| `NOW_QUIC_MAX_STREAMS`  | `1024`  | 认证后 QUIC 双向流的最大并发数。                     |
| `NOW_TCP_DATA_BUF_SIZE` | `32768` | TCP 转发中每个方向的缓冲区大小。                     |
| `NOW_UDP_DATA_BUF_SIZE` | `65536` | UDP 目标套接字（target-socket）的接收缓冲区大小。    |
| `NOW_TCP_DIAL_TIMEOUT`  | `15s`   | TCP 目标连接超时时间。                               |
| `NOW_UDP_DIAL_TIMEOUT`  | `15s`   | UDP 目标连接超时时间。                               |
| `NOW_TCP_READ_TIMEOUT`  | `30s`   | TCP 连接某一方向达到 EOF（文件结束）后的宽限期。     |
| `NOW_UDP_IDLE_TIMEOUT`  | `120s`  | QUIC 连接及 QUIC DATAGRAM/UoT 数据流的空闲超时时间。 |
| `NOW_HANDSHAKE_TIMEOUT` | `5s`    | 添加抖动（jitter）前的基础认证截止时间。             |
| `NOW_REPORT_INTERVAL`   | `5s`    | 本地检查点（checkpoint）事件的触发间隔。             |
| `NOW_SHUTDOWN_TIMEOUT`  | `5s`    | 关机期间等待端点（endpoint）进入空闲状态的超时时间。 |
| `NOW_RELOAD_INTERVAL`   | `3600s` | 重新加载 PEM 证书的最小尝试间隔。                    |

持续时间（Duration）配置项支持 `500ms`、`15s` 和 `2m` 等格式。无效值将回退使用上述默认值。整数类型的配置项必须为非负数。

`NOW_SERVICE_COOLDOWN` 也存在于运行时默认配置中，当前默认值为 `3s`；该参数保留供服务端重试逻辑使用。

## 关机流程

接收到 `SIGINT` 信号时，Portal 将启动优雅关机流程：

1. 停止接受新连接的循环（accept loops）。
2. 关闭 QUIC 端点及所有活跃连接。
3. 等待端点进入空闲状态，最长等待时间为 `NOW_SHUTDOWN_TIMEOUT`。
4. 在相同的关机时限内，等待 TCP 连接任务完成。
5. 重置活跃的速率限制器（rate limiters）。
6. 输出关机完成日志并刷新日志缓冲区。

## 运维建议

- 对于长期运行的公网部署，建议设置 `tls=2`。
- 将证书和密钥文件存放在代码仓库之外，并限制文件权限，仅允许服务运行用户访问。
- 生产环境建议使用 `log=event` 以便于机器解析；在发布/部署期间建议使用 `log=info`。
- 当部署仅需单一入口传输协议时，建议使用 `net=tcp` 或 `net=udp`。
- 若服务运行在仅支持特定地址族（address family）的进程管理器（supervisor）之后，建议显式指定监听地址。
- 每次修改配置后，请务必验证实际生效的启动 URL。
- 请将 `rate` 和 `etar` 视为进程级的公平性控制手段，而非针对单个客户端的配额限制。
