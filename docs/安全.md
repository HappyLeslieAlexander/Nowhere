# 安全须知

Nowhere v1 的设计规模虽小，但它仍属于暴露在网络中的服务。
本文档概述了协议线路格式（wire format）之外的安全特性及运维人员可配置的选项。

## 支持的传输层安全机制

Portal 仅支持基于 TLS 的模式：

| 模式    | 行为                                            |
| ------- | ----------------------------------------------- |
| `tls=1` | 启动时在内存中生成自签名证书。                  |
| `tls=2` | 从 `crt` 和 `key` 加载 PEM 格式的证书链及私钥。 |

不支持明文传输（即 `tls=0` 模式）。

对于 TLS/TCP 和 QUIC，均禁用了 TLS 1.3 的 Early Data（早期数据）和 Half-RTT Server Data（半 RTT 服务器数据）功能。系统不接受作为 0-RTT 数据提交的应用层数据。

TLS/TCP 承载常规 TCP 转发和 UoT UDP 数据流。QUIC 在双向流（bidirectional streams）上承载 TCP 转发，并在 DATAGRAM 帧中承载 UDP 数据流。UoT 的建立请求及数据包帧，仅在通过与常规 TCP 转发相同的 TLS/TCP 认证流程后才会被接受。

## 共享密钥

共享密钥即 URL 中的用户名部分（经百分号解码后）。该密钥不能为空，且长度不得超过 255 字节（UTF-8 编码）。

请使用高熵值密钥。切勿复用在其他系统中使用的用户密码。切勿将生产环境密钥记录在 shell 历史记录、会广泛暴露命令行参数的进程管理器、问题追踪系统、示例代码或代码仓库文件中。

协议派生逻辑如下：

```text
auth_key = SHA-256(shared_key_bytes)
```

共享密钥不会改变协议规范定义的字段顺序或填充（padding）布局，仅会影响认证标签（authentication tags）。

## 协议规范与 ALPN 协商

通信双方必须就以下内容达成一致：

- 共享密钥；
- `effective_spec`（有效协议规范）；
- `effective_alpn`（有效 ALPN）；以及
- 帧版本 `1`。

`spec` 会改变认证常量、填充方式及字段顺序；`alpn` 仅影响 TLS 和 QUIC 的协商过程。若 ALPN 不匹配，传输层建立将失败；若共享密钥或协议规范不匹配，认证过程将失败。

## 认证失败时的行为

在认证成功之前，不会转发任何目标流量。失败
身份验证路径会延迟至一个采样后的绝对截止时间点：

```text
NOW_HANDSHAKE_TIMEOUT * [0.8, 1.2]
```

延迟结束后，TCP 连接在没有应用层响应的情况下关闭。QUIC 连接关闭时，应用层代码为 `0x01`，原因为 `access denied`（拒绝访问）。详细的失败信息仅写入本地日志，且仅在发起网络连接关闭操作之后才会写入。

## 身份验证前的资源限制

Portal 在接受客户端之前会限制未经身份验证的处理任务：

- QUIC 连接在进入身份验证阶段前，必须先经过 QUIC Retry（重试）流程。
- 整个进程范围内最多允许 256 个未经身份验证的连接。
- 每个 IPv4 `/32` 或 IPv6 `/64` 地址段最多允许 32 个未经身份验证的连接。
- 身份验证前收到的 QUIC DATAGRAM 帧会被丢弃（drained），并受限于 64 KiB 的总保留配额。

这些限制旨在减轻未经身份验证连接带来的资源压力，但不能替代网络级过滤或服务监控机制。

## TLS 信任策略

对于 `tls=1`，客户端必须显式信任或固定（pin）生成的证书。该生成的证书在重启后不会保持不变。

对于 `tls=2`，除非有意采用不同的信任模型，否则客户端应使用标准的平台证书验证和服务器名称检查。在受控部署环境中，指纹固定（fingerprint pinning）可能很有用，但当证书变更时，必须有计划地轮换指纹。

## 部署指南

- 对于公网或长期运行的部署，请使用 `tls=2`。
- 确保证书和密钥文件仅对运行服务的用户可读。
- 尽可能将 Portal 部署在防火墙规则之后，该规则应匹配预期的客户端集合。
- 如果不需要某种入站传输协议，建议显式指定 `net=tcp` 或 `net=udp`。即使指定 `net=tcp`，仍可通过 UoT（UDP over TCP）支持 UDP 流量。
- 监控 `CHECK_POINT` 计数器和进程重启情况。
- 通过协调客户端和服务器的部署更新来轮换共享密钥。
- 将调试日志视为敏感信息，因为它们可能泄露操作细节。
- 当管理层状态、API 响应、事件流和日志包含 Portal URL 时，应将其视为敏感信息，因为 URL 中的用户名即为共享密钥。 ## 非目标范围

该门户（Portal）不提供以下功能：

- 用户账户；
- 密钥轮换协议消息；
- 远程管理 API；
- 应用层授权规则；
- 目标允许列表；或
- 明文传输。

若部署场景需要上述控制功能，请在门户之外进行实现。
[OpenCtrl](https://github.com/NodePassProject/OpenCtrl) 是一个受支持的外部管理层；
有关其安全性和流程生命周期边界的信息，请参阅[集成指南](integrations.md)。
