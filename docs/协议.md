# Nowhere 协议 v1

## 1. 状态与范围

本文档定义了 Nowhere 代理协议的第 1 版，以及参考实现所需的 Portal 配置。

Nowhere 通过 TLS/TCP 或 QUIC 承载经过身份验证的 TCP 和 UDP 代理流量。TCP 流量使用专用的 TLS/TCP 连接或 QUIC 双向流进行传输；UDP 流量则使用 QUIC DATAGRAM 帧或带有长度前缀的 UDP-over-TCP (UoT) 数据流。每个传输连接均通过共享密钥进行身份验证，同时利用确定性的 `spec` 值来确定身份验证形式、填充方式及字段布局。

文中的关键词 **MUST**（必须）、**MUST NOT**（不得）、**REQUIRED**（必需）、**SHOULD**（应）、**SHOULD NOT**（不应）和 **MAY**（可以）应被视为规范性要求。

## 2. 约定

- 除非另有说明，所有整数域均为无符号类型，并采用网络字节序。
- `u8`、`u16` 和 `u64` 分别表示 1 字节、2 字节和 8 字节的无符号整数。
- `||` 表示无分隔符的字节拼接。
- 文本转换为字节时使用 UTF-8 编码。
- 长度是指经过 URL 百分号解码（percent decoding）后的字节数。
- `SHA-256(x)` 表示 `x` 的 32 字节 SHA-256 摘要。
- `HMAC-SHA256(k, x)` 表示使用密钥 `k` 和消息 `x` 进行的 HMAC-SHA256 运算。
- `HKDF-Extract(salt, ikm)` 和 `HKDF-Expand(prk, info, length)` 使用 HKDF 规范中定义的 SHA-256 算法。引号内的标签表示其字面 ASCII 字节值，不包含结尾的零字节（null byte）。

## 3. Portal 配置

Portal 通过一个 URL 进行配置：

```text
portal://<shared-key>@<listen-host>:<listen-port>?tls=<mode>&spec=<spec>&alpn=<alpn>&net=<mode>&dial=<ip-or-auto>&rate=<mbps>&etar=<mbps>&crt=<path>&key=<path>&log=<level>
```

最小配置：

```text
portal://secret@:2077
```

URL 中的用户名即为共享密钥（shared key）。不支持密码部分，且必须拒绝包含密码的配置。必须指定监听端口和非空的共享密钥。未知的查询参数将被忽略。

### 3.1 输入规则

共享密钥（shared key）、`spec` 和 `alpn` 均按 UTF-8 格式进行百分号解码。`spec` 或 `alpn` 中的字面字符 `+` 保持不变，不会被转换为空格。如果查询参数中出现重复的键，则仅使用第一次出现的值。

| 输入项   | 要求               | 解码后的 UTF-8 字节长度 |
| -------- | ------------------ | ----------------------- |
| 共享密钥 | 必填且不能为空     | `1..255`                |
| `spec`   | 可选；空值视为省略 | 非空时为 `1..255`       |
| `alpn`   | 可选；空值视为省略 | 非空时为 `1..255`       |

### 3.2 参数

| 参数   | 默认值  | 语义                                                         |
| ------ | ------- | ------------------------------------------------------------ |
| `tls`  | `1`     | `1` 表示生成自签名证书。`2` 表示从 `crt` 和 `key` 加载 PEM 文件。其他值无效。 |
| `spec` | `auto`  | 用于 v1 认证、填充（padding）和帧布局的确定性种子。          |
| `alpn` | `now/1` | QUIC/TLS ALPN 覆盖值。不改变其他任何协议字段。               |
| `net`  | `mix`   | 选择入站传输协议。`tcp` 启用 TLS/TCP，`udp` 启用 QUIC/UDP，`mix` 同时启用两者。缺失或空值时默认为 `mix`；其他值无效。 |
| `dial` | `auto`  | 用于出站 TCP 和 UDP 套接字的本地 IP 字面值。空值、无效值、主机名或 `auto` 均表示使用操作系统默认值。 |
| `rate` | `0`     | 客户端到目标端的速率限制（单位：Mbps）。                     |
| `etar` | `0`     | 目标端到客户端的速率限制（单位：Mbps）。                     |
| `crt`  | 空      | `tls=2` 模式下使用的 PEM 证书链。                            |
| `key`  | 空      | `tls=2` 模式下使用的 PEM 私钥。                              |
| `log`  | `info`  | `none`、`debug`、`info`、`warn`、`error` 或 `event`。若指定了未知值，则默认采用 `info`。 |

`rate` 和 `etar` 接受正十进制整数。若设为零、负数、无效值或未指定该参数，则相应的限制功能将被禁用。换算公式如下：

```text
bytes_per_second = mbps * 125000
```

速率限制器由单个 Portal 进程处理的所有会话共享。它分别为“客户端到目标”和“目标到客户端”的流量维护独立的令牌桶。

`net` 参数并不用于选择代理流量的类型。TLS/TCP 支持普通 TCP 转发和 UoT；QUIC 支持基于双向流的 TCP 转发以及基于 DATAGRAM 帧的 UDP 转发。

### 3.3 监听地址

若监听主机（listen host）为空，则针对 `net` 选定的每种传输方式，在同一端口上分别绑定 IPv4 和 IPv6 通配符套接字。`0.0.0.0` 仅绑定 IPv4，`[::]` 仅绑定 IPv6。若指定 IP 字面量，则绑定该特定地址。若指定主机名，则对其进行解析并使用解析出的第一个地址。在 Portal 开始接收流量之前，所有选定的套接字必须均已完成绑定。

## 4. 有效协议规范 (Effective Protocol Spec)

在建立会话之前，通信双方必须构建相同的有效协议规范。

```text
effective_spec = 若第一个 `spec` 值非空，则为其解码后的值；否则为 "auto"
effective_alpn = 若第一个 `alpn` 值非空，则为其解码后的值；否则为 "now/1"
```

`effective_spec` 和 `effective_alpn` 各自发挥独立作用。更改 `spec` 不会改变 ALPN；更改 `alpn` 也不会改变身份验证、填充或帧布局。

共享密钥也不会改变由规范决定的结构形态；它仅用于派生身份验证密钥。 ### 4.1 HKDF 根（Root）

```text
spec_bytes = UTF8(effective_spec)
spec_salt  = SHA-256(spec_bytes)
spec_prk   = HKDF-Extract(spec_salt, spec_bytes)

derive(label, length) = HKDF-Expand(spec_prk, UTF8(label), length)
```

v1 版本定义了以下输出：

```text
spec_id               = base64url-no-pad(derive("spec id", 8))
auth_magic            = derive("auth magic", 8)
auth_info             = derive("auth hmac info", 32)
auth_context          = derive("auth context", 32)
auth_layout_seed      = derive("auth frame layout", 8)
proxy_layout_seed     = derive("proxy frame layout", 8)
auth_padding_len_seed = derive("auth padding length", 2)
auth_padding_key      = derive("auth padding key", 32)
tcp_padding_len_seed  = derive("tcp request padding length", 1)
tcp_padding_key       = derive("tcp request padding key", 32)
```

`spec_id` 是一个用于诊断和一致性验证的标识符。它不会在 v1 协议帧中进行传输。

### 4.2 字段顺序推导

v1 版本使用确定性的 Fisher-Yates 洗牌算法。给定数组 `A`、种子（seed）以及初始种子偏移量：

```text
for i from len(A) - 1 down to 1:
k = initial_offset + (len(A) - 1 - i)
j = seed[k] mod (i + 1)
swap A[i] and A[j]
```

三种布局的推导方式如下：

| 布局                  | 初始数组                           | 种子偏移量                   |
| --------------------- | ---------------------------------- | ---------------------------- |
| 认证 (Authentication) | `[magic, nonce, padding, tag]`     | `auth_layout_seed` 中的 `0`  |
| TCP 请求              | `[version, target, padding]`       | `proxy_layout_seed` 中的 `0` |
| UDP 头部              | `[version, type, flow_id, target]` | `proxy_layout_seed` 中的 `1` |

如果认证布局的洗牌操作导致初始数组未发生变化（即顺序保持不变），则结果**必须**向左循环移位一次，变为 `[nonce, padding, tag, magic]`。TCP 和 UDP 布局不执行此循环移位操作。

所有对等端（peers）**必须**严格实现这些算法。推导过程**不得**依赖共享密钥、挂钟时间、随机性、区域设置（locale）、平台整数位宽、映射（map）迭代顺序或传输库的默认设置。

## 5. 传输与 TLS

Nowhere v1 支持基于 TCP 的 TLS 1.3 和基于 UDP 的 QUIC。这两种传输方式使用相同的证书，并仅通告一个 ALPN 值：`effective_alpn`。

对于 QUIC，Portal 节点：

- 仅通告一个 ALPN 值：`effective_alpn`；
- 启用 QUIC DATAGRAM；
- 在接受连接之前，要求通过 QUIC Retry 机制进行地址验证；
- 使用 BBR 拥塞控制算法；
- 初始通告一个双向流（bidirectional stream）和 64 KiB 的连接级接收信用额度（receive credit）；
- 认证完成后，将双向流限制提高至 `NOW_QUIC_MAX_STREAMS`，并将连接级接收信用额度提高至 32 MiB； - 宣告每个流（stream）拥有 16 MiB 的接收信用额度（receive credit）；
- 允许每个连接存在最多 32 MiB 未确认的流数据；
- 不接受单向流；
- 使用 `NOW_UDP_IDLE_TIMEOUT` 作为 QUIC 空闲超时时间；以及
- 不发送传输层的保活（keepalive）数据包。

对于 QUIC 和 TLS/TCP，Portal 均明确禁用了 TLS 1.3 早期数据（early data）和半 RTT（half-RTT）服务器数据。因此，应用数据绝不会作为 0-RTT 数据被接受。

QUIC 客户端必须仅提供一个等于 `effective_alpn` 的 ALPN 值，启用 QUIC DATAGRAM，并支持双向流。TCP 客户端必须协商使用相同 ALPN 的 TLS 1.3。ALPN 不匹配将导致连接建立失败。不支持明文 TCP。

### 5.1 TLS 模式

当 Portal 启动时，`tls=1` 会为 `localhost` 生成一个新的自签名证书。在此模式下，是否信任该证书由应用程序决定，且必须进行明确确认。

`tls=2` 会从 `crt` 加载 PEM 格式的证书链，并从 `key` 加载 PEM 格式的私钥。Portal 启动时，这两个文件必须均有效。在处理 `ClientHello` 握手请求时，如果已达到 `NOW_RELOAD_INTERVAL` 设定的时间间隔，Portal 会重新加载这些文件。重新加载成功后，缓存的证书将被更新；若重新加载失败，系统会记录日志，并继续使用原有的证书。

使用 `tls=2` 的客户端应执行常规的平台证书验证及服务器名称验证。TLS 信任策略和 SNI 不会改变 v1 版本的线路格式（wire format）。

## 6. 连接生命周期

QUIC 握手完成后：

1. 客户端打开第一个双向流作为认证流。
2. 客户端写入且仅写入一个 v1 认证帧，并结束其发送端。
3. 认证成功后，后续的每个双向流均可承载一个 TCP 转发会话。
4. 在已认证连接的生命周期内，UDP 转发流量可使用 QUIC DATAGRAM 帧。

Portal 在 QUIC 或 TLS 握手完成后确定一个绝对认证截止时间。该截止时间为 `NOW_HANDSHAKE_TIMEOUT` 乘以一个 `[0.8, 1.2]` 范围内的系统随机因子；若无法获取随机数，则直接使用配置的超时时间。对于 QUIC，等待认证流建立及读取其完整帧的过程共用这同一个截止时间。在认证成功之前，不会转发任何目标流量。

认证前到达的 QUIC DATAGRAM 帧会被主动排空（drained）。Portal 最多保留总计 64 KiB 的数据，并在认证成功后将这些保留的帧交付给会话。超出此限制的帧以及因认证失败而保留的所有帧均会被丢弃。

对于 TLS/TCP，每个代理流使用一个新连接：

1. 客户端完成 TLS 1.3 握手。
2. 客户端发送且仅发送一个 v1 认证帧。
3. 客户端立即发送一个 v1 TCP 请求帧，或者先将已认证连接保持空闲并置于客户端侧的预热连接池（warm pool）中。
4. 普通目标选择 TCP 转发，剩余所有字节均为客户端到目标的原始 TCP 载荷。保留目标 `uot.nowhere.invalid:0` 选择 UoT 模式，随后客户端继续执行第 9.2 节定义的建立流程及数据包帧传输。

在 TCP 传输层上，不存在认证响应、转发后的连接复用或多路复用机制。连接池中的连接一旦被首次请求使用，即告消耗，且仅承载一个 TCP 转发会话或一个 UoT 流。每个被接受的 TCP 连接相对于其他连接而言均是无状态的。若已认证连接在 40 秒内未发送请求帧，Portal 将关闭该连接。

参考实现客户端会在冷连接池（cold pool）上启动一个预热连接。使用一个连接会启动最多两个替换连接，同时确保空闲连接与正在进行的连接总数不超过配置的连接池上限。每个未被使用的连接槽位在创建 30 秒后过期；过期不会触发替换操作。

## 7. 身份验证

凭证密钥（credential key）为：

```text
auth_key = SHA-256(shared_key_bytes)
```

客户端提供一个 32 字节的随机数（nonce）。对于每个连接，客户端**应**生成一个新的、具有密码学安全性的随机数。

### 7.1 填充与标签

```text
auth_padding_len = 1 + (u16be(auth_padding_len_seed) mod 255)

auth_padding = HKDF-Expand(
auth_padding_key,
UTF8("auth padding bytes") || nonce || u8(auth_padding_len),
auth_padding_len
)

auth_tag = HMAC-SHA256(
auth_key,
auth_info || auth_context || nonce ||
u8(auth_padding_len) || auth_padding
)
```

`auth_padding_len` 的取值范围为 `1..255`。

### 7.2 帧

身份验证帧按以下顺序包含这四个元素：

| 元素      | 编码                                  |
| --------- | ------------------------------------- |
| `magic`   | `auth_magic` (8 字节)                 |
| `nonce`   | 32 字节                               |
| `padding` | `auth_padding_len_u8 || auth_padding` |
| `tag`     | `auth_tag` (32 字节)                  |

完整帧的长度为 `74..328` 字节。接收方**必须**验证帧长度、magic 值、声明的填充长度、确定性填充字节以及 HMAC 标签。接收方还**必须**要求帧之后紧接着出现流结束（end-of-stream）信号。标签和填充的比较**应**采用恒定时间（constant-time）算法。

若身份验证成功，则立即继续后续操作。若出现流缺失、帧截断、EOF、缺少 FIN、尾随字节，或字段、填充、HMAC 无效等情况，系统将延迟处理直至达到采样绝对截止时间（sampled absolute deadline）。随后，参考 Portal 将使用应用代码 `0x01` 和原因 `access denied` 关闭 QUIC 连接，或在无应用响应的情况下关闭 TCP 连接。网络关闭操作将在详细故障信息写入 Portal 本地日志之前发起。服务关闭操作会取消延迟。

Portal 针对 TCP 和 QUIC 实施统一的进程级预认证准入限制：总连接数上限为 256 个，且针对每个 IPv4 `/32` 或 IPv6 `/64` 网段的上限为 32 个。若 QUIC 连接尝试导致任一上限超标，该尝试将被静默忽略；若 TCP 连接导致任一上限超标，该连接将被立即关闭。一旦认证成功或失败，相应的连接槽位即会释放。

## 8. TCP 转发

每个 TCP 转发会话使用一个新的双向 QUIC 流或一个专用的 TLS/TCP 连接。在这两种情况下，客户端都会先写入相同的请求帧，紧接着发送原始的客户端到目标（client-to-target）TCP 字节流。

### 8.1 请求填充

```text
tcp_padding_len = tcp_padding_len_seed[0] mod 64

tcp_padding = HKDF-Expand(
tcp_padding_key,
UTF8("tcp request padding bytes") || target_utf8 ||
u8(tcp_padding_len),
tcp_padding_len
)
```

`tcp_padding_len` 的取值范围为 `0..63`。对于特定的 `(effective_spec, target)` 组合，其填充字节内容是确定性的。

### 8.2 请求帧

请求帧包含以下三个元素，按指定的 TCP 顺序排列：

| 元素      | 编码方式                            |
| --------- | ----------------------------------- |
| `version` | `u8(1)`                             |
| `target`  | `target_len_u16 || target_utf8`     |
| `padding` | `tcp_padding_len_u8 || tcp_padding` |

接收方必须拒绝非 `1` 的版本号、无效的目标地址、错误的填充长度或错误的填充字节。请求填充内容不会被转发给目标。

解析请求后，Portal 会解析并连接到目标地址，并可选择将出站套接字绑定到 `dial`。随后，它在两个方向上转发字节流。当一个方向达到 EOF（流结束）时，另一个方向最多可继续传输 `NOW_TCP_READ_TIMEOUT` 时长。

## 9. UDP 转发

UDP 转发有两种特定于传输协议的形式：

- QUIC DATAGRAM：在单个已认证的 QUIC 连接上，通过 `(flow_id, target)` 对数据流进行多路复用。
- UoT：在单个已认证的 TLS/TCP 连接上，将目标数据流承载为带有长度前缀的数据包。

这两种形式共用目标验证、出站 UDP 拨号、速率限制、空闲超时和 UDP 计数器机制。除此之外，它们的线路帧格式是相互独立的。 ### 9.1 QUIC 数据报（DATAGRAM）头部

每个 QUIC 数据报包含一个按特定顺序排列的头部，随后是透明载荷（opaque payload）。

| 元素      | 编码                                    |
| --------- | --------------------------------------- |
| `version` | `u8(1)`                                 |
| `type`    | 下列值之一                              |
| `flow_id` | `u64`（作用域限定于已认证的 QUIC 连接） |
| `target`  | `target_len_u16 || target_utf8`         |

头部元素按特定的 UDP 顺序排列。载荷始终紧跟在完整头部之后，且不会被重排。

| 类型             | 值   | 方向与含义                                                   |
| ---------------- | ---- | ------------------------------------------------------------ |
| Request（请求）  | `1`  | 客户端至 Portal；建立或复用流，并将载荷转发至目标（target）。 |
| Response（响应） | `2`  | Portal 至客户端；接收自目标的载荷。                          |
| Close（关闭）    | `3`  | 客户端至 Portal；关闭匹配的流。任何载荷均被忽略。            |

在单个 QUIC 连接内，UDP 流由 `(flow_id, target)` 标识。具有相同流 ID 但目标不同的流被视为不同的流。响应使用请求中的流 ID 和目标。Portal 会在 `NOW_UDP_IDLE_TIMEOUT` 后关闭非活跃流。

格式错误的数据报、不受支持的版本、未知类型以及 Portal 接收到的响应帧均不会被转发。针对未知流的关闭帧不产生任何效果。

### 9.2 UDP-over-TCP (UoT)

UoT 仅在经过身份验证的 TLS/TCP 连接上可用。通过发送第 8 节中定义的标准 TCP 请求帧（使用以下保留的目标地址）来启用该功能：

```text
uot.nowhere.invalid:0
```

该保留目标地址用于协议切换，**不得**被视为 TCP 目标地址。其请求帧采用与其他所有 TCP 请求相同的字段顺序和确定性填充方式。

发送该请求后，客户端**必须**立即发送且仅发送一个设置帧（setup frame）：

```text
target_len_u16 || target_utf8
```

`target_len_u16` 的值必须在 1 到 512 之间，且 `target_utf8` 必须符合第 10 节的规定。Portal（网关）读取完整设置目标地址的操作受 `NOW_HANDSHAKE_TIMEOUT` 限制。随后，它会解析目标地址并打开一个已连接的 UDP 套接字，并可选择根据 `dial` 参数绑定源地址。

设置完成后，双向通信仅包含数据包帧（packet frames）：

```text
payload_len_u16 || payload
```

`payload_len_u16` 的值在 0 到 65535 之间。每个帧恰好对应一个 UDP 数据包，因此实现必须保留帧与数据包的边界。UoT 不包含流 ID、消息类型或带内关闭帧。一条 TLS/TCP 连接承载一个目标流；客户端针对不同的并发目标使用独立的连接。

任一方向的流量都会刷新 `NOW_UDP_IDLE_TIMEOUT` 计时器。正常的 TCP EOF（流结束）、截断或无效的帧结构、目标套接字错误、空闲超时或服务关闭都会导致 UoT 流终止。有效载荷字节数计入 `rate` 和 `etar`，并记录在 UDP 计数器中。双字节数据包长度、设置帧、身份验证帧和请求帧不计入 UDP 有效载荷。

## 10. 目标地址编码

TCP 请求、QUIC DATAGRAM 头部和 UoT 设置帧使用相同的目标地址表示形式。`target_utf8` 必须：

- 是有效的 UTF-8 编码； - 字节长度在 `1` 到 `512` 之间；
- 包含非空的端口部分；以及
- 对 IPv6 字面量使用方括号括起来，例如 `[2001:db8::1]:443`。

未加方括号的目标地址（target）只能包含用于分隔主机和端口的单个冒号。v1 编解码器验证不要求主机部分非空，也不将端口解析为整数；即使帧被接受，目标地址解析和连接建立仍可能失败。

## 11. 速率限制与计数器

Portal 实施一个进程级的限制器，包含两个独立的方向：

- `rate`：从客户端到目标地址的 TCP 字节数以及 QUIC DATAGRAM/UoT UDP 载荷字节数；
- `etar`：从目标地址到客户端的 TCP 字节数以及 QUIC DATAGRAM/UoT UDP 载荷字节数。

速率限制不会选择或修改 QUIC 拥塞控制算法。BBR 保持不变。

在启动时以及随后的每个 `NOW_REPORT_INTERVAL` 周期，当活动日志级别包含事件时，Portal 会发出以下事件记录：

```text
CHECK_POINT|MODE=0|PING=0ms|POOL=<n>|TCPS=<n>|UDPS=<n>|TCPRX=<bytes>|TCPTX=<bytes>|UDPRX=<bytes>|UDPTX=<bytes>
```

| 字段    | 含义                                  |
| ------- | ------------------------------------- |
| `TCPS`  | 活动的 TCP 转发流。                   |
| `UDPS`  | 活动的 QUIC DATAGRAM 和 UoT 流。      |
| `TCPRX` | 从客户端到目标地址的 TCP 字节数。     |
| `TCPTX` | 从目标地址到客户端的 TCP 字节数。     |
| `UDPRX` | 从客户端到目标地址的 UDP 载荷字节数。 |
| `UDPTX` | 从目标地址到客户端的 UDP 载荷字节数。 |

`POOL` 表示正在等待首个请求帧的已认证 TLS/TCP 连接数量。TLS 握手、正在进行的认证、活动的 TCP 或 UoT 转发以及 QUIC 连接不计入其中。`MODE` 和 `PING` 保持为 v1 中所示的值。 `NOW_REPORT_INTERVAL` 仅控制此处的本地遥测上报周期，不控制 QUIC 保活流量。

## 12. 运行时控制参数

这些环境变量用于控制参考实现中的 Portal 组件，不会改变 v1 派生逻辑或帧格式。

| 变量名                  | 默认值  | 用途                                                         |
| ----------------------- | ------- | ------------------------------------------------------------ |
| `NOW_QUIC_MAX_STREAMS`  | `1024`  | QUIC 双向流的最大并发数。                                    |
| `NOW_TCP_DATA_BUF_SIZE` | `32768` | TCP 转发各方向的缓冲区大小。                                 |
| `NOW_UDP_DATA_BUF_SIZE` | `65536` | UDP 目标套接字接收缓冲区大小。                               |
| `NOW_TCP_DIAL_TIMEOUT`  | `15s`   | TCP 目标连接超时时间。                                       |
| `NOW_UDP_DIAL_TIMEOUT`  | `15s`   | UDP 目标连接超时时间。                                       |
| `NOW_TCP_READ_TIMEOUT`  | `30s`   | TCP 单向传输结束后保留的宽限期。                             |
| `NOW_UDP_IDLE_TIMEOUT`  | `120s`  | QUIC 空闲超时时间，以及 QUIC DATAGRAM/UoT 流的空闲超时时间。 |
| `NOW_HANDSHAKE_TIMEOUT` | `5s`    | 包含抖动（jitter）的认证截止时间的基准值。                   |
| `NOW_REPORT_INTERVAL`   | `5s`    | 本地事件上报周期。                                           |
| `NOW_SHUTDOWN_TIMEOUT`  | `5s`    | 关机期间等待端点空闲的超时时间。                             |
| `NOW_RELOAD_INTERVAL`   | `3600s` | 重新加载 PEM 证书的最小间隔时间。                            |

持续时间（Duration）参数支持 Portal 可识别的格式，例如 `500ms`、`15s` 或 `2m`。若设置了无效值，将使用表中列出的默认值。整数参数必须为非负数；若设置了无效值或负数，将使用表中列出的默认值。

## 13. 互操作性要求

两个对等端仅在满足以下条件相同时才能互操作：

- 共享密钥（shared key）；
- `effective_spec` 和 v1 派生规则；
- `effective_alpn`；以及
- 帧版本 `1`。

ALPN 不匹配会导致 TLS 或 QUIC 协商失败。若 ALPN 相同但共享密钥或有效规范（effective spec）不同，则会在处理代理流量之前发生认证失败。

符合规范的实现必须拒绝格式错误或被截断的帧、无效的字段长度、不支持的版本、不正确的确定性填充、无效的认证标签、认证流末尾的额外字节，以及无效的 UoT 设置帧或数据包帧。实现必须根据本文档中的限制来约束所有内存分配。

v1 派生规则、标签、字段顺序算法、整数编码和帧类型值均为协议常量。更改其中任何一项均与 Nowhere v1 不兼容。

## 14. 一致性

实现应至少验证以下情况：

1. 省略、留空以及显式指定 `spec=auto` 都会产生相同的有效规范（effective spec）。
2. 省略或留空 `alpn` 均会选择 `now/1`。
3. 显式指定 ALPN 仅改变 `effective_alpn`。
4. 接受解码后为 255 字节的输入，拒绝 256 字节的输入。
5. 使用相同规范但不同共享密钥时，产生相同的派生布局，但认证标签不同。
6. 不同的规范产生各自的认证常量、填充和布局。
7. 认证、TCP 请求、UDP 数据报、UoT 设置和 UoT 数据包的编码器与解码器能够进行往返互操作（即编码后解码能还原）。
8. 拒绝错误的版本、目标长度、填充长度、填充字节、帧类型和标签。
9. 若有效帧之后存在任何字节，则拒绝该认证流。
10. 针对 `uot.nowhere.invalid:0` 的 TLS/TCP 请求会切换到 UoT 模式，保留双向的 UDP 数据包边界，并将该流记录为 UDP 流量。对于以下固定输入：

```text
shared key = "secret"
spec       = "auto"
ALPN       = "now/1"
nonce      = 32 字节，每个字节均为 0x07
TCP target = "example.com:443"
```
