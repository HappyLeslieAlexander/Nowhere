# 集成

Nowhere 将协议服务内置于 Portal 中，并支持与外部管理及客户端软件进行组合。这些集成是可选的：Portal 既可以作为独立的前台进程使用，兼容的客户端也可以直接实现[协议规范](protocol.md)。

## OpenCtrl 管理

[OpenCtrl](https://github.com/NodePassProject/OpenCtrl) 是一个受支持的控制平面，用于管理由 URL 定义的运行时进程。其进程契约与 Nowhere 相匹配：被管理的二进制程序接收一个配置 URL，保持在前台运行，并通过标准输出报告日志和指标。

OpenCtrl 增加了持久化实例定义、生命周期操作、带版本控制的 REST API、服务器发送事件（SSE）以及基于检查点（checkpoint）的计数器功能。它不会更改 Portal 的配置或网络传输协议。

### 启动控制平面

构建或安装这两个二进制程序，然后将 OpenCtrl 指向 Nowhere 可执行文件：

```sh
openctrl 'master://127.0.0.1:8080?bin=/usr/local/bin/nowhere'
```

首次启动时，OpenCtrl 会输出一个 API 密钥。请使用该密钥进行受保护的 API 请求：

```sh
BASE='http://127.0.0.1:8080/api/v2'
API_KEY='<openctrl-api-key>'
```

若需构建可通过网络访问的控制平面，请根据 OpenCtrl 的文档配置 TLS 并限制访问权限。

### 创建 Portal

通过将完整的 Portal URL 作为实例的 `url` 参数传入来创建一个实例：

```sh
curl -X POST "${BASE}/instances" \
-H "X-API-Key: ${API_KEY}" \
-H 'Content-Type: application/json' \
-d '{"alias":"edge-a","url":"portal://change-me@:2077?spec=nightfall"}'
```

OpenCtrl 会异步启动子进程。因此，在进程达到 `running`（运行中）状态之前，创建操作的响应可能会显示为 `stopped`（已停止）。保存返回的 `id`，
然后读取该实例或订阅事件以获取实时状态：

```sh
INSTANCE_ID='<instance-id>'

curl -H "X-API-Key: ${API_KEY}" \
"${BASE}/instances/${INSTANCE_ID}"

curl -N -H "X-API-Key: ${API_KEY}" \
"${BASE}/events"
```

该示例特意省略了 `log` 参数。Nowhere 默认的 `info` 级别会保留
常规操作日志并输出 EVENT 检查点。

对于长期运行的 Portal，请使用稳定的证书和明确的公开
配置：

```text
portal://change-me@:2077?tls=2&crt=/etc/nowhere/cert.pem&key=/etc/nowhere/key.pem&spec=nightfall&alpn=now%2F1&net=mix
```

证书和私钥的路径由 Nowhere 子进程解析，
且必须可由运行 OpenCtrl 的操作系统用户读取。

### 生命周期操作

OpenCtrl 通过 `PATCH` 请求接收生命周期操作指令：

```sh
curl -X PATCH "${BASE}/instances/${INSTANCE_ID}" \
-H "X-API-Key: ${API_KEY}" \
-H 'Content-Type: application/json' \
-d '{"action":"restart"}'
```

可根据需要将 `restart` 替换为 `start` 或 `stop`。`reset` 操作会清除 OpenCtrl 暴露的计数器，但不会更改 Portal 配置：

```sh
curl -X PATCH "${BASE}/instances/${INSTANCE_ID}" \
-H "X-API-Key: ${API_KEY}" \
-H 'Content-Type: application/json' \
-d '{"action":"reset"}'
```

删除实例并停止其管理的进程：

```sh
curl -X DELETE "${BASE}/instances/${INSTANCE_ID}" \
-H "X-API-Key: ${API_KEY}"
```

### EVENT 指标

Nowhere 会在 Portal 启动时以及随后每隔 `NOW_REPORT_INTERVAL`（默认值为 5 秒）发送一条检查点（checkpoint）记录：

```text
CHECK_POINT|MODE=0|PING=0ms|POOL=<n>|TCPS=<n>|UDPS=<n>|TCPRX=<bytes>|TCPTX=<bytes>|UDPRX=<bytes>|UDPTX=<bytes>
```

OpenCtrl 会处理这些记录，而不会将其作为普通日志事件转发。这些字段在实例资源中的对应关系如下：

| EVENT 字段 | OpenCtrl 字段 | Nowhere 中的含义                    |
| ---------- | ------------- | ----------------------------------- |
| `MODE`     | `mode`        | v1 版本中固定为 `0`。               |
| `PING`     | `ping`        | v1 版本中固定为 `0ms`。             |
| `POOL`     | `pool`        | 等待首个请求的已认证 TLS/TCP 连接。 |
| `TCPS`     | `tcps`        | 活动的 TCP 转发流。                 |
| `UDPS`     | `udps`        | 活动的 QUIC DATAGRAM 和 UoT 流。    |
| `TCPRX`    | `tcprx`       | 客户端至目标端的 TCP 字节数。       |
| `TCPTX`    | `tcptx`       | 目标端至客户端的 TCP 字节数。       |
| `UDPRX`    | `udprx`       | 客户端至目标端的 UDP 载荷字节数。   |
| `UDPTX`    | `udptx`       | 目标端至客户端的 UDP 载荷字节数。   |

字节计数器针对单个 Portal 实例在进程范围内统计。OpenCtrl 在受管进程重启期间维护累积计数，并可独立于子进程重置其公开的统计总数。

请保持 OpenCtrl 默认的 5 秒上报间隔。在接收到第一个检查点（checkpoint）后，如果超过 15 秒未收到后续检查点，OpenCtrl 会将其视为错误。因此，将 `NOW_REPORT_INTERVAL` 设置为 15 秒或更长可能会导致误报健康状态变更。

### 运行边界

- OpenCtrl 将子进程日志中包含 `ERROR` 字符串的任何普通行视为错误信号。由于 Nowhere 也使用 `ERROR` 标识连接级故障，因此握手或中继失败可能会导致实例状态暂时变为 `error`，尽管 Portal 本身仍在运行。下一个有效的检查点会将状态恢复为 `running`。
- 在 Unix 系统上，OpenCtrl 使用 `SIGTERM` 信号停止子进程。Nowhere 目前未针对 `Ctrl-C`/`SIGINT` 实现优雅关闭逻辑；默认的 `SIGTERM` 行为会立即退出，而不会执行 Portal 显式的排空（drain）和刷新（flush）流程。
- Portal URL 中包含共享密钥。OpenCtrl 会持久化该 URL，通过 REST 和 SSE 实例对象返回它，并将其记录在进程启动日志中。请务必保护好 API 密钥、状态目录、日志文件以及所有管理客户端。
- 单个 OpenCtrl 主进程（master）为其管理的实例使用相同的配置 `bin` 路径，且子进程会继承主进程的环境变量。若实例间存在差异，建议优先使用 Portal URL 参数进行区分。如果实例需要不同的 `NOW_*` 环境变量值，请使用独立的 OpenCtrl 主进程或受信任的包装程序（wrapper）。
- 端口分配由运维人员负责。每个 Portal 实例必须拥有互不冲突的 TCP 和/或 UDP 监听地址。

## Anywhere 客户端

[Anywhere](https://github.com/NodePassProject/Anywhere) 是一款支持 iOS、iPadOS 和 tvOS 平台的原生客户端。其 Nowhere 实现支持在两种传输协议族上进行 TCP 转发和 UDP 流量传输：

| Anywhere `net` | Portal 监听器 | TCP 流量     | UDP 流量               |
| -------------- | ------------- | ------------ | ---------------------- |
| `udp`          | QUIC/UDP      | 双向 QUIC 流 | QUIC 数据报 (DATAGRAM) |
| `tcp`          | TLS/TCP       | 专用认证连接 | UoT 连接               |

使用 `net=mix` 配置的 Portal 接受上述任一客户端模式。若 Portal 仅限 `net=tcp` 或 `net=udp`，则客户端必须使用相应的 Anywhere 模式。

### 匹配 Portal 与客户端 URL

使用稳定证书运行公共 Portal：

```text
portal://change-me@:2077?tls=2&crt=/etc/nowhere/cert.pem&key=/etc/nowhere/key.pem&spec=nightfall&alpn=now%2F1&net=mix
```

在 Anywhere 中导入 QUIC/UDP 客户端配置：

```text
nowhere://change-me@relay.example.com:2077?net=udp&spec=nightfall&sni=relay.example.com&alpn=now%2F1#Edge
```

或者选择带有预热连接池的 TLS/TCP 模式：

```text
nowhere://change-me@relay.example.com:2077?net=tcp&pool=5&spec=nightfall&sni=relay.example.com&alpn=now%2F1#Edge
```

Portal 和客户端必须使用相同的共享密钥、`spec` 和 ALPN。客户端配置中的主机地址应为可从外部访问的 Portal 地址，而非服务端 URL 中使用的空通配符主机名。

### 配置映射

| 设置项                        | Portal URL                         | Anywhere URL                                                 |
| ----------------------------- | ---------------------------------- | ------------------------------------------------------------ |
| Shared key                    | URL 用户名                         | URL 用户名；必须完全匹配。                                   |
| Address                       | 监听主机（可留空）                 | 可访问的服务器主机名或 IP 地址。                             |
| `net`                         | 启用 `tcp`、`udp` 或 `mix` 监听器  | 为此客户端选择 `tcp` 或 `udp`。                              |
| `spec`                        | 协议形态种子 (Protocol-shape seed) | 必须解析为相同值；两者默认均为 `auto`。                      |
| `alpn`                        | TLS/QUIC ALPN                      | 必须解析为相同值；两者默认均为 `now/1`。                     |
| `sni`                         | 不使用                             | 证书服务器名称；默认为客户端主机名。                         |
| `pool`                        | 不使用                             | TLS/TCP 预热连接池大小 `0..9`；仅在 `net=tcp` 时有效，默认 `5`，设为 `0` 则禁用。 |
| `tls`, `crt`, 查询参数 `key`  | 选择并配置 Portal 证书             | 不导出；Anywhere 始终使用 TLS 并应用其信任策略。             |
| `dial`, `rate`, `etar`, `log` | Portal 运行时控制                  | 不包含在客户端分享链接中。                                   |

共享密钥（Shared key）即 URL 中的用户名。请勿将其与 Portal 的查询参数 `key` 混淆，后者是 `tls=2` 模式下 PEM 格式私钥的路径。

### 证书信任

对于公网或长期部署，请使用 `tls=2` 并配合稳定的证书。
Anywhere 既可以验证标准的平台信任链，也可以验证用户通过指纹明确信任的证书。
关于 Portal 端的信任模型，请参阅[安全说明](security.md)。

`tls=1` 模式下的 Portal 证书是自签名的，且在每次进程启动时重新生成。
因此，重启后保存的指纹将不再匹配。
仅在受控测试场景下使用此模式，即明确信任当前证书，或有意启用了 Anywhere 的全局“允许不安全连接”（Allow Insecure）设置。
切勿将“允许不安全连接”作为公网部署的信任模型。

## 更多集成

该协议规范独立于 OpenCtrl 和 Anywhere。额外的核心与客户端集成可以在不修改 Portal 的情况下，实现相同的身份验证、传输及帧协议契约。
